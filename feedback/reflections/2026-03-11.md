## Reflection — 2026-03-11 session 3 (SPIRE003)

**Topic**: General

### What went well
- The overall TDD workflow executed smoothly: scaffold → coverage matrix → test cases → implementation → review → fix → verify. All 56 tests pass, full suite (216) has no regressions.
- Spawning 4 test-case-writer agents in parallel was efficient — all 4 completed within ~2.5 minutes and produced correct files.
- The test-researcher agent produced a thorough 55-case coverage matrix with good categorization (explicit default(T), bare default literal, struct variants, exclusions).
- The code reviewer caught a real issue (IsInsideIsPattern not walking through IConstantPatternOperation) and a test gap (missing NoReport_IsPattern_DefaultLiteral). Both were addressed.
- The IOperation-based detection strategy (`OperationKind.DefaultValue`) was clean — a single operation kind covers both `default(T)` and bare `default`, making the analyzer simple.

### What didn't go well
- **Implementer agent ran out of budget.** It spent excessive turns debugging 3 test failures caused by `using` directives in case files ending up after type declarations from `_shared.cs`. The agent went down a long investigation path (considering creating an OperationViewer tool, hypothesizing about Roslyn recovery modes) instead of recognizing the root cause quickly: `using` directives in case files are placed after `_shared.cs` content when combined, violating C# syntax rules. The fix was trivial — move the usings to `_shared.cs`.
- **Implementer agent used `cat` to write files** instead of Write/Edit tools. The user noticed and called this out. This is a tool discipline issue — the agent should use the dedicated tools.
- **The IsInsideIsPattern bug existed in the initial implementation.** The implementer should have checked the IOperation tree for `is default(T)` patterns more carefully. The `IDefaultValueOperation` is wrapped in `IConstantPatternOperation` before reaching `IIsPatternOperation` — this is predictable from the AST structure.
- **4 fixes needed after the implementer finished**: 3 test case file fixes (using directives) + 1 analyzer fix (IsInsideIsPattern). The lead had to do all 4 inline.

### Suggestions
- **Add `using System.Collections.Generic;` and `using System.Threading.Tasks;` to `_shared.cs` by default** in the new-rule skill's scaffolding step. These are common enough that most rules will need them, and placing them in `_shared.cs` avoids the combined-source ordering issue. This would have prevented 3 of the 4 post-implementation fixes.
- **Add a warning to test-case-writer agent instructions**: "Do NOT add `using` directives in case files. All usings must be in `_shared.cs` because case file content is appended after `_shared.cs` type declarations — a `using` after a type declaration is a C# syntax error." This is a non-obvious constraint of the test system.
- **Add a convention to `.claude/rules/test-conventions.md`**: Document that `using` directives must go in `_shared.cs`, not in individual case files, due to the concatenation order.
- **Implementer agent instructions could include a checklist**: Before reporting completion, verify that `IsInsideEqualityComparison` and `IsInsideIsPattern` (or similar exclusion methods) walk through all intermediate IOperation wrappers (conversions, constant patterns, etc.). The IOperation tree often has wrapper nodes between the target operation and the context operation.
- **Budget monitoring**: The implementer agent burned most of its budget on a problem that took the lead 4 lines to fix. Consider adding to implementer instructions: "If you've spent more than 3 turns on a single failing test, message the lead with a clear description of the problem instead of continuing to investigate."

## Reflection — 2026-03-11 session 3 (continued — test infrastructure refactor)

**Topic**: General

### What went well
- The user identified a fundamentally better solution to the `_shared.cs` + case file concatenation problem. Instead of patching symptoms (moving usings around), we changed the compilation model to use separate syntax trees — which is how Roslyn is designed to work.
- The refactor was clean and minimal: `AnalyzerTestBase.LoadAndParse` and `GetAnalyzerDiagnosticsAsync` were the only methods that needed changes. The offset computation was deleted entirely.
- `global using` in `_shared.cs` was the right mechanism to share imports across trees — no per-case-file boilerplate needed.
- All 217 tests passed on the first try after the refactor.
- The `/update-docs` skill worked well — cross-reference map correctly identified all 5 files that needed updates (plus MEMORY.md), and the grep for stale references confirmed nothing was missed.

### What didn't go well
- I initially proposed the wrong fix (suggestion 1 from the reflection: "add common usings to `_shared.cs` by default"). The user correctly pointed out that passing multiple files to Roslyn is the real solution. I should have considered this earlier — using separate compilation units is standard Roslyn practice.
- The implementer agent used `cat` with heredocs instead of the Write/Edit tools. The user noticed and called it out. This is a recurring tool discipline problem with spawned agents.

### Suggestions
- **Add tool discipline to agent instructions**: Add a constraint to `.claude/agents/analyzer-implementer.md` and `.claude/agents/test-case-writer.md`: "Use the Write tool to create files and the Edit tool to modify files. Do NOT use `cat`, heredocs, `echo`, or `sed` via Bash for file operations."
- **The `docs/test-case-format.md` could include a brief note about `global using`**: Something like "If your case file needs a namespace not covered by `_shared.cs`, add a regular `using` at the top of your case file. To add a shared import for all cases, add a `global using` to `_shared.cs`." This was added in this session but could be more explicit with an example.
- **Consider adding a compilation-error check to `AnalyzerTestBase`**: Before running the analyzer, check `compilation.GetDiagnostics()` for errors and fail the test with a clear message if the test code doesn't compile. Currently, compilation errors in test code are silent — the analyzer just doesn't fire, leading to confusing "expected diagnostic but got none" failures. This would have made the original `using`-after-type-declaration bug obvious immediately.
